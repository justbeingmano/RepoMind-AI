## Blocking Issues

> These issues would prevent the system from starting or functioning correctly if triggered.

### B-01 â€” Root-Level Module Imports Require `uvicorn` to Be Launched from Project Root

**Files:** `backend/core/semantic_loader.py`, `backend/core/profile_loader.py`

**Description:**
Both files import root-level modules using bare names:
```python
from repo_utils import is_github_repository, resolve_full_name
from semantic_hybrid_recommender import SemanticHybridRecommender
from smart_profile_recommender_v2 import UserProfile
```

These imports only work if the **current working directory is the project root** when `uvicorn` is launched.

**Trigger condition:** Running `uvicorn` from inside a subdirectory (e.g., `cd backend && uvicorn main:app`).

**Error:** `ModuleNotFoundError: No module named 'repo_utils'`

**Workaround:** Always run from the project root:
```bash
uvicorn backend.main:app --reload --port 8000
```

**Fix Required:** Not fixing per policy (would require restructuring imports).

---

### B-02 â€” `processed.json` Must Exist Before Starting the Backend

**Files:** `backend/core/semantic_loader.py`, `backend/core/profile_loader.py`

**Description:**
Both `load_semantic_hybrid()` and `load_profile_recommender()` load `processed.json` on the **first API request**. If the file doesn't exist, the server starts fine but crashes on the first request.

**Error:**
```
FileNotFoundError: Data file not found: .../processed.json
```

**Trigger condition:** Fresh clone without running `process.py` first.

**Workaround:** Ensure `processed.json` is present (either committed to the repo or generated by `python process.py`).

---

### B-03 â€” `sentence-transformers` Must Be Installed for Semantic Search

**Files:** `core/search_engine.py` â†’ `_load_embedding_model()`

**Description:**
If `sentence-transformers` is not installed, the search engine fails on first request with a clear ImportError:

```python
raise ImportError(
    "sentence-transformers is required for semantic search.\n"
    "Install it with: pip install sentence-transformers numpy"
)
```

**Workaround:** `pip install sentence-transformers` (already in `requirements.txt`).

## Warning Issues

> These issues do not block execution but may cause degraded or unexpected behavior.

### W-01 â€” Dual Index Directories May Cause Confusion

**Files:** `backend/core/semantic_loader.py`, `core/search_engine.py`

**Description:**
The backend uses `VECTOR_DB_PATH = ROOT / "vector_db"` while the CLI default for `search_engine.py` is `vector_db_path="storage"`. This means:
- Running `python core/search_engine.py --query "..."` builds/uses `storage/`
- Running the FastAPI backend builds/uses `vector_db/`

Both directories may have different (or stale) indexes.

**Impact:** Stale or mismatched indexes are silently used if fingerprints match. Results may differ between CLI and API.

**Mitigation:** The fingerprint system detects dataset changes and rebuilds automatically. Only causes confusion during development.

---

### W-02 â€” `search_index/` Contains Pickle Files (Stale/Incompatible)

**Files:** `search_index/bm25.pkl`, `search_index/repos.pkl`, etc.

**Description:**
The `search_index/` directory uses Python pickle format from what appears to be an older version of the system. These files are not used by the current backend. If any future code mistakenly loads them, there may be schema incompatibility.

**Impact:** Not currently used. No active risk.

**Classification:** Deprecated storage.

**Description:**
This file is very large (~27 KB) and appears to be an earlier version of `smart_profile_recommender_v2.py`. It is not imported by any active module in `backend/`. If it was accidentally imported, it could produce different results.

**Impact:** None currently. Risk is accidental future import.

### W-04 â€” `print()` Debug Statement in Production Code

**Files:** `smart_profile_recommender_v2.py` line 553

**Description:**
```python
print(f"\n[DEBUG] Cleaned/expanded query terms: {clean_terms}\n")
```

This debug print executes on every `search_with_profile()` call (i.e., every `/profile/search` and related API request). It will pollute server logs.

**Impact:** Minor log noise in production. No functional impact.

---

### W-05 â€” Bare `except:` Clauses in `scraper.py`

**Files:** `scraper.py` lines 65, 84, 183

**Description:**
```python
except:
    return None
```

Bare `except` clauses catch all exceptions including `KeyboardInterrupt`, `SystemExit`, and memory errors. This can mask unexpected failures silently.

**Impact:** Scraper may silently skip repos due to unexpected errors without any logged reason.

### W-08 â€” Vite Proxy URL Double-`/api` Behavior

**Files:** `frontend/src/api/client.js`, `frontend/vite.config.js`

**Description:**
The Vite proxy rewrites `/api â†’ ""` (strips `/api`). The frontend `client.js` calls:
```javascript
api.post('/api/advisor/explain', ...)
```
where `api` has `baseURL = /api`. So the actual request URL is `/api/api/advisor/explain`. The proxy strips the first `/api`, leaving `/api/advisor/explain`, which matches the FastAPI route `prefix="/api/advisor"`.

This works **correctly** but is non-obvious and could break if either the proxy rewrite rule or the `baseURL` is changed without updating the other.

**Impact:** Currently functional. Risk during future refactoring.
